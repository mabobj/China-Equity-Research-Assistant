# current-phase.md

## 文档目标

本文档用于约束 **当前阶段** 的开发范围，帮助 Codex 和人工协作时始终聚焦在最近 1~2 个里程碑上，避免无序扩展。

本文件应短、小、硬，只描述：

- 当前正在做什么
- 当前明确不做什么
- 当前阶段的成功标准是什么
- 下一阶段预告是什么

## 当前项目阶段

当前处于：

**Phase 4-B：选股器深筛版 / 单票研究与策略能力向全市场候选聚合过渡阶段**

项目已经完成的基础能力：

1. 项目骨架、脚本、测试入口
2. A 股基础数据接入
3. 单票技术分析底层
4. 单票研究输入（公告列表、财务摘要）
5. 结构化单票研究报告
6. 结构化单票交易策略
7. 全市场规则初筛选股器

当前阶段的核心目标是：

> 把“初筛候选”进一步变成“可操作候选”，也就是对候选股票自动聚合研究与策略结果，形成深筛列表。

---

## 当前阶段必须完成的能力

### 1. 深筛选股聚合能力

基于已有的：

- screener 初筛结果
- research 报告
- strategy 计划

生成统一的深筛候选对象。

每个深筛候选至少应包含：

- symbol
- name
- base_list_type
- base_rank
- base_screener_score
- research_action
- research_overall_score
- research_confidence
- strategy_action
- strategy_type
- ideal_entry_range
- stop_loss_price
- take_profit_range
- review_timeframe
- thesis
- short_reason
- priority_score

### 2. 深筛 API

新增清晰、可直接前端接入的深筛接口，例如：

- `GET /screener/deep-run`

应支持最小必要参数，例如：

- `max_symbols`
- `top_n`
- `deep_top_n`

### 3. 深筛排序与优先级

第一版必须使用：

- 简单规则
- 可解释逻辑
- 可测试实现

不要引入黑盒模型，不要引入 LLM。

### 4. 为前端接入做准备

这一阶段结束时，深筛结果 schema 必须足够清晰，以便下一阶段前端能够直接展示：

- 候选分层
- 简要 thesis
- 关键价格区间
- 关键风险与复查时间

## 当前阶段明确不做的内容

本阶段禁止扩展到以下方向：

### 1. 不做新外部数据源接入

不要新增：

- 新闻 provider
- 公告正文/PDF 解析
- 新财务数据源
- 新舆情平台抓取

### 2. 不做 LLM 深度总结

不要新增：

- OpenAI 调用
- 长文研究报告
- 自由文本投资结论
- 自动多 agent 辩论

### 3. 不做个性化持仓策略

不要引入：

- 持仓成本
- 仓位历史
- 组合暴露
- 个性化卖出策略

这些应放到后续“交易记录 / 持仓上下文”阶段。

### 4. 不做前端复杂页面

当前阶段不要：

- 做复杂图表
- 做前端筛选交互体系
- 做复杂状态管理
- 做页面视觉优化

### 5. 不做定时任务系统增强

不要在这一阶段实现：

- 定时全市场扫描调度
- 自动日更
- 自动夜间复盘任务

## 当前阶段的工程原则

### 1. 先聚合，后美化

当前重点是把：

- 初筛结果
- 研究结果
- 策略结果

通过统一 schema 聚合起来。

不是优先生成更长的文字。

### 2. 先结构，后智能

输出必须以结构化对象为核心。

短文案只允许做补充字段，例如：

- `thesis`
- `short_reason`

### 3. 先可测，后扩展

深筛相关能力必须有：

- pipeline 测试
- 聚合逻辑测试
- API 测试

且测试不能依赖实时外网。

### 4. 个股失败不能让全局失败

深筛过程中：

- 某只股票 research 失败
- 某只股票 strategy 失败
- 某只股票数据不完整

都不应让整次 deep run 崩掉。

---

## 当前阶段成功标准

满足以下条件，即视为当前阶段完成：

1. 能从初筛结果中选出需要深筛的股票
2. 能对这些股票自动聚合研究与策略
3. 能输出统一、清晰、结构化的深筛列表
4. 深筛结果支持后续前端直接展示
5. 测试通过且不依赖实时外网
6. README 同步更新接口说明
