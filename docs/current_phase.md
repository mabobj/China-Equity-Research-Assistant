# 当前阶段

## 1. 文档目标

本文只回答当前版本最重要的四个问题：

1. 现在优先做什么
2. 现在明确不做什么
3. 为什么当前重点这样定
4. 当前阶段完成的标准是什么

## 2. 当前阶段定义

当前项目处于：

**v2.2 准备期：长期主线对齐与数据底座收口阶段**

这意味着：

- `workspace-bundle + workflow + trade/review` 产品壳层已经成型
- `prediction / backtest / evaluation` 最小主线已经接入
- 当前不再以“补更多页面或模块”为主目标
- 当前优先把文档、架构口径、数据底座和长期方向重新对齐

## 3. 当前阶段必须完成的事

### 3.1 长期方向文档收口

必须做到：

- README、架构、路线图、当前阶段文档不再讲不同版本的故事
- 明确项目长期主线是“因子发现、验证、组合与监控系统”
- 明确当前产品工作台只是这条长期主线的前台壳层

### 3.2 数据底座面向长期方向继续收口

当前必须重点盯住的是“足够支撑长期方向”的几项关键缺口，而不是无止尽优化：

- 点时一致性
- 点时一致性第一阶段已完成，第二阶段已把 `workspace-bundle`、关键单票 route 与 workflow 的 `as_of_date` 读取口径收拢为“可显式指定、不可安全历史重算时受控失败”
- 关键数据域扩展规划
- 关键数据域补齐第一阶段已完成，当前已经把 `基准目录 / 行业与板块分类 / 市场广度 / 基础风险代理` 收成标准 schema、日级数据产品与最小只读接口
- 关键数据域补齐第二阶段已完成，当前已经补上 `基准按日日线` 主链路，并把风险代理从“仅由广度派生”推进为“广度 + 主基准收益/趋势”的保守组合代理
- 复权与公司行为口径
- 复权与公司行为口径前两阶段已完成，当前已把日线价格口径与公司行为元数据显式接入 schema、normalize、local store 与 API 输出，并支持 `raw / qfq / hfq` 显式请求与分口径存储
- 数据血缘与版本可追溯
- provider 能力矩阵与健康度清晰化
- provider 能力矩阵与健康度第二阶段已完成，当前已存在集中 capability 策略、主用/兜底角色说明、capability 级健康判定以及最小只读诊断 API

### 3.3 预测与评估主线继续保留

当前阶段不重做预测主线，但要明确它在长期方向里的位置：

- 不是独立新系统
- 而是未来因子验证与模型评估的中间层
- 逐步服务于研究、选股、交易和复盘

## 4. 当前阶段明确不做的事

本阶段不进入：

- 自动下单
- 券商接入
- 高频交易
- 新一轮大规模前端重构
- 新的无边界 provider 扩张
- 复杂持仓引擎与组合归因
- 调度系统、队列系统、DAG 平台重构

## 5. 为什么当前重点这样定

原因很简单：

1. 当前系统已经具备“可用的产品壳层”，但长期方向还没有在文档和数据底座上完全收口。
2. 如果这时继续无序扩功能，会让未来的因子验证和预测评估主线越来越难接。
3. 长期因子系统真正需要的不是更多零散页面，而是更稳定的数据标准、更可重建的样本、更清晰的阶段边界。

## 6. 当前阶段成功标准

满足下面几点，就可以认为当前阶段完成：

1. 文档体系统一回答：
   - 项目长期目标是什么
   - 当前已经完成什么
   - 下一步优先做什么
2. 数据底座的长期缺口被明确列出，并且范围受控
3. 团队后续开发默认以长期方向为主线，而不是回到早期 MVP 叙事
4. `taskbook-v2.1`、`roadmap.md`、`architecture.md`、`README.md` 的角色边界清晰

## 8. 当前专项：选股工作台稳定性修复

当前阶段除继续推进 `taskbook-v2.2` 外，增加一个受控的稳定性专项：

- 不改初筛评分规则；
- 不改 research / strategy 主链路；
- 不做新的前端重构；
- 只修复 `/screener` 在真实批量执行中的稳定性与可观测性问题。

本专项遵循下面的边界：

1. 初筛只消费已有可用数据，不在用户点击运行时临时补全全量财务和公告；
2. 公共底座数据由日级准备阶段沉淀；
3. 个股重数据在深筛或单票研究阶段按需补全；
4. 长任务状态必须可恢复、可刷新、可诊断。

当前优先顺序为：

1. 修复 workflow stale 误判，确保长任务不在读状态时“消失”；
2. 拆分 active-run 与 latest-batch 首屏读取链路，降低刷新超时概率；
3. 增强结构化日志，补齐 run 级、批次级、单票级诊断信息；
4. 在稳定性恢复后，再继续做批量初筛执行提速。

当前进度：

- 第一阶段“运行态不丢失”已完成：
  - 后端已修正读状态时的 stale 误判；
  - `/screener/active-run` 轻量查询入口已落地；
  - 前端刷新时会优先恢复当前运行中的初筛任务。
- 第二阶段“刷新链路减负”已完成：
  - 已新增 `/screener/latest-batch-summary` 轻量摘要接口；
  - 已新增 `/screener/latest-batch/results` 延迟结果接口；
  - 首屏不再默认重建整窗结果，也不在刷新时同步触发预测字段补齐；
  - `/screener` 前端已改为“摘要先到、结果后到”的两段式加载。
- 第三阶段“日志与诊断增强”已完成：
  - `workflow / pipeline / batch service` 已补齐 run 级、批次级、单票级结构化日志；
  - 当前可以通过 `run_id` 串联查看启动、选股范围、heartbeat、单票完成、批次落盘与窗口摘要读取过程；
  - 单票日志已能区分 `skipped / failed_placeholder / candidate` 三类结果，并给出各数据域耗时与质量状态。
- 第四阶段“批量执行提速”已完成：
  - 初筛继续明确“只消费已有可用数据”的边界，不在批量执行时逐票远端补全财务摘要与公告窗口；
  - `market_data_service` 已为 `financial_summary / announcements` 补齐 `allow_remote_sync=False` 支持，本地无数据时返回受控降级结果，不再阻塞整批；
  - `screener pipeline` 已引入受控小并发（默认 4 worker），在保持评分规则与结果契约不变的前提下提升吞吐；
  - 已补性能相关回归测试，验证 provider 参数透传、缺失降级与并发扫描的兼容性。
