# 当前阶段

## 1. 文档目标

本文只回答当前版本最重要的四个问题：

1. 现在优先做什么
2. 现在明确不做什么
3. 为什么当前重点这样定
4. 当前阶段完成的标准是什么

## 2. 当前阶段定义

当前项目处于：

**v2.2 准备期：长期主线对齐与数据底座收口阶段**

这意味着：

- `workspace-bundle + workflow + trade/review` 产品壳层已经成型
- `prediction / backtest / evaluation` 最小主线已经接入
- 当前不再以“补更多页面或模块”为主目标
- 当前优先把文档、架构口径、数据底座和长期方向重新对齐

## 3. 当前阶段必须完成的事

### 3.1 长期方向文档收口

必须做到：

- README、架构、路线图、当前阶段文档不再讲不同版本的故事
- 明确项目长期主线是“因子发现、验证、组合与监控系统”
- 明确当前产品工作台只是这条长期主线的前台壳层

### 3.2 数据底座面向长期方向继续收口

当前必须重点盯住的是“足够支撑长期方向”的几项关键缺口，而不是无止尽优化：

- 点时一致性
- 点时一致性第一阶段已完成，第二阶段已把 `workspace-bundle`、关键单票 route 与 workflow 的 `as_of_date` 读取口径收拢为“可显式指定、不可安全历史重算时受控失败”
- 关键数据域扩展规划
- 复权与公司行为口径
- 数据血缘与版本可追溯
- provider 能力矩阵与健康度清晰化
- provider 能力矩阵与健康度第二阶段已完成，当前已存在集中 capability 策略、主用/兜底角色说明、capability 级健康判定以及最小只读诊断 API

### 3.3 预测与评估主线继续保留

当前阶段不重做预测主线，但要明确它在长期方向里的位置：

- 不是独立新系统
- 而是未来因子验证与模型评估的中间层
- 逐步服务于研究、选股、交易和复盘

## 4. 当前阶段明确不做的事

本阶段不进入：

- 自动下单
- 券商接入
- 高频交易
- 新一轮大规模前端重构
- 新的无边界 provider 扩张
- 复杂持仓引擎与组合归因
- 调度系统、队列系统、DAG 平台重构

## 5. 为什么当前重点这样定

原因很简单：

1. 当前系统已经具备“可用的产品壳层”，但长期方向还没有在文档和数据底座上完全收口。
2. 如果这时继续无序扩功能，会让未来的因子验证和预测评估主线越来越难接。
3. 长期因子系统真正需要的不是更多零散页面，而是更稳定的数据标准、更可重建的样本、更清晰的阶段边界。

## 6. 当前阶段成功标准

满足下面几点，就可以认为当前阶段完成：

1. 文档体系统一回答：
   - 项目长期目标是什么
   - 当前已经完成什么
   - 下一步优先做什么
2. 数据底座的长期缺口被明确列出，并且范围受控
3. 团队后续开发默认以长期方向为主线，而不是回到早期 MVP 叙事
4. `taskbook-v2.1`、`roadmap.md`、`architecture.md`、`README.md` 的角色边界清晰

## 7. 下一阶段预告

在当前阶段结束后，下一阶段建议进入：

重点围绕：

- 数据域补齐
- 特征与标签资产化
- 回测与评估继续增强
- 为真正的因子验证层打地基
