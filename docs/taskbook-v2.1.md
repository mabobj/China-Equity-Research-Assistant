# 项目任务书（v2.1，产品化增强版）

## 1. 目标定义

本阶段目标不是继续堆模块，而是同时完成两条主线：

- 功能主线：在现有 `workspace-bundle + workflow + trade/review` 基础上，新增“预测与评估”能力链。
- 体验主线：把当前工作台从“工程控制台”升级为“消费级可用产品”，让用户知道先看什么、先做什么、为什么这么做。

统一成功标准：

- 能看懂：用户进入页面后 30 秒内理解当前结论和下一步动作。
- 能执行：用户可在 3 分钟内完成一次“选股 -> 单票判断 -> 记录交易”。
- 能复盘：任一交易都能回溯当时判断、数据质量、执行偏差与复盘结论。

## 2. 当前基线（以代码为准）

- 单票主入口：`GET /stocks/{symbol}/workspace-bundle`
- 选股主入口：`POST /workflows/screener/run` + `GET /workflows/runs/{run_id}`
- 深筛主入口：`POST /workflows/deep-review/run` + `GET /workflows/runs/{run_id}`
- 交易与复盘入口：`/trades` 与 `/reviews` 已可用，不再是预留页
- 数据清洗层：`bars`、`financial_summary`、`announcements` 已接入主链路
- 质量消费：`/research` 与 `/screener` 已开始消费 `quality_status`

## 3. 核心问题诊断

- 页面信息层级混乱：关键动作与调试信息混在同一层，用户不知先后顺序。
- 术语负担过高：页面直接暴露 `runtime_mode`、`fallback`、`strategy_alignment` 等内部术语。
- 操作路径过长：单票页与选股页都存在“同页过多任务”问题。
- 文案存在历史残留：个别页面或文档仍保留“预留/未启用”旧叙事。

## 4. 产品原则（v2.1 必须遵守）

- 先结论，再证据，后细节。
- 默认展示业务语言，技术细节下沉到“高级信息”。
- 所有长任务必须有进度感知与阶段反馈。
- 局部失败不阻断主任务，失败影响要可解释。
- 术语在页面、接口、文档保持同义一致。

## 5. 范围与非范围

本阶段纳入：

- 单票、选股、交易、复盘四个现有页面的信息架构重构
- 预测评估主线的后端基础能力建设与渐进接入
- 文档、术语、操作手册的统一收口

本阶段不纳入：

- 自动下单与券商接入
- 复杂持仓引擎与组合归因
- 队列/DAG/调度平台重构
- 新增大量页面种类或重型前端框架

## 6. 双轨任务拆分

### 6.1 功能轨（预测与评估）

阶段 F1：点时特征与标签底座

- 新增 `dataset_service` 与 `label_service`
- 建立 `feature_version` 与 `label_version`
- 产出日级点时特征与标签数据集

阶段 F2：回测与评估底座

- 新增 `backtest_service` 与 `evaluation_service`
- 建立 walk-forward 回测流程
- 先跑规则基线，再跑模型基线

阶段 F3：预测服务 MVP

- 新增 `prediction_service` 与 `experiment_service`
- 输出 `predictive_score`、`model_confidence`、`model_version`
- 结果先用于离线评估，再渐进接入在线链路

阶段 F4：接回现有产品链

- `workspace-bundle` 增加 `predictive_snapshot`（可选字段）
- `screener` 排序引入预测分（保留质量门控）
- `trade/review` 记录绑定模型版本与预测分

### 6.2 体验轨（消费级改造）

阶段 U1：首页与导航收口

- 首页入口改为任务导向：看单票、跑选股、记交易、做复盘
- 去除“预留/未启用”旧叙事
- 统一术语卡与入口说明

阶段 U2：单票页重构

- 重排为“结论卡 -> 证据卡 -> 行动卡 -> 详情抽屉”
- `fallback/runtime/quality` 默认折叠
- 冲突提示改写为用户可理解语言

阶段 U3：选股页重构

- 结构改为“运行状态 -> 最新批次 -> 结果表 -> 单股详情”
- workflow 参数分层，默认只保留高频参数
- 深筛入口折叠到高级区

阶段 U4：交易与复盘重构

- 交易页优先“快速记录”，高级参数渐进展示
- 复盘页优先“待复盘案例”，突出对照关系
- 一致性冲突提示需明确“原判断、当前动作、冲突点”

## 7. 可执行任务包

### 包 1（当前包）：认知收口包

- 修正首页与文档中的历史残留文案
- 发布 v2.1 任务书
- 建立术语统一映射并同步到 README/manuals
- 前端最小检查：`type-check + lint`

### 包 2：单票体验重构包

- 拆分 `stock-workspace` 组件
- 首屏只保留决策主内容
- 高级运行信息下沉折叠
- 单票 smoke 测试补充

### 包 3：选股体验重构包

- 拆分 `screener-workspace` 组件
- 结果视图优先于参数视图
- 运行反馈与失败提示重写
- 选股 smoke 测试补充

### 包 4：交易复盘体验收口包

- 交易记录表单简化与冲突引导优化
- 复盘详情对照视图强化
- 最小统计区块上线
- `/trades`、`/reviews` 契约测试补充

### 包 5：预测底座包

- 上线特征、标签、回测、评估服务骨架
- 建立版本化产物 schema
- 回测结果输出契约固定

### 包 6：预测接入包

- `workspace-bundle` 接入预测快照
- `screener` 排序融合预测分
- trade/review 写入预测版本元数据

## 8. 验收标准

体验验收：

- 新用户完成“选股 -> 单票 -> 交易”主路径成功率达到 85% 以上
- 单票首屏可理解性显著提升，关键动作可在 30 秒内定位
- 页面错误提示可回答“发生了什么、影响什么、下一步做什么”

功能验收：

- 点时特征与标签可重建、可追溯、可版本化
- 回测流程可重复运行并输出稳定指标
- 预测结果可被单票与选股链路消费且不破坏现有主流程

稳定性验收：

- 局部模块失败仍返回可用主结果
- fallback/runtime 可见字段持续可解释
- 新增测试不依赖实时外网

## 9. 风险与应对

- 风险：功能开发压过体验收口  
  应对：双轨里程碑绑定，未达体验验收不得进入下一功能包。

- 风险：术语继续漂移  
  应对：建立术语清单并在页面与文档同源复用。

- 风险：预测接入破坏现有链路  
  应对：先可选字段接入，保留现有逻辑为回退路径。

## 10. 协作方式

- 每个任务包必须包含：代码改动、测试改动、文档改动。
- 每个任务包结束输出：变更摘要、验收结果、残留风险。
- 优先做“用户可感知价值”最高的改动，再做技术扩展。
