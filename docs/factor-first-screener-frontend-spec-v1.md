# 因子优先初筛前端交互规格 v1

## 1. 文档目标

本文件定义因子优先初筛第一阶段的前端交互主线、页面分区、状态管理边界和验收标准。

目标不是做完整策略编辑器，而是让前端在现有项目基础上，稳定承接：

- 方案选择
- 方案驱动运行
- 结果查看
- 方案级反馈回看

## 2. 设计原则

### 2.1 方案优先

用户进入初筛页后，首先看到的是“当前方案是什么”，而不是先看到一批缺乏上下文的结果。

### 2.2 结果不再脱离方案

所有结果视图都必须能回答：

- 来自哪套方案
- 哪个版本
- 当时关键配置是什么

### 2.3 第一阶段不做重型编辑器

第一阶段只做：

- 选方案
- 看方案摘要
- 跑方案
- 看结果
- 看复盘摘要

不做：

- 原子因子自由编辑器
- DSL 规则搭建器
- 拖拽式策略编排器

## 3. 页面主线

初筛页主线固定为：

1. 方案
2. 运行
3. 结果
4. 复盘

不建议再以“结果列表是唯一中心”来组织页面。

## 4. 页面区块

### 4.1 方案区

职责：

- 选择当前方案
- 显示当前版本
- 显示关键因子组、阈值摘要、质量门控摘要

最少展示字段：

- `scheme_name`
- `scheme_version`
- `description`
- `selected_factor_groups`
- `threshold_summary`
- `quality_gate_summary`

第一阶段交互：

- 选择方案
- 查看方案详情
- 查看版本摘要

不强制第一阶段支持网页内编辑方案。

### 4.2 运行区

职责：

- 发起一次方案运行
- 展示 run 状态
- 展示批次摘要

最少展示字段：

- `run_id`
- `batch_id`
- `scheme_id`
- `scheme_version`
- `status`
- `started_at`
- `candidate_count`

第一阶段运行参数：

- `batch_size`
- `max_symbols`
- `force_refresh`

这些运行参数可以保留，但必须与方案摘要区分显示。

### 4.3 结果区

职责：

- 展示候选列表与 bucket 分布
- 展示单只股票结果详情
- 显示该结果对应的方案上下文

单只结果详情至少显示：

- `scheme_name`
- `scheme_version`
- `bucket`
- `screener_score`
- `trend_score`
- `predictive_score`
- `quality_summary`
- `top_factor_reasons`

如果某字段当前后端尚未提供，应在文档和 UI 中明确为“待接入”，而不是前端自行推断。

### 4.4 复盘区

职责：

- 展示方案历史 runs
- 展示基础转化与反馈统计

最少展示：

- 历史运行列表
- `total_runs`
- `total_candidates`
- `entered_research_count`
- `trade_count`
- `review_count`
- `bucket_counts`

## 5. 组件拆分建议

建议从现有 `frontend/src/components/screener-workspace.tsx` 中拆分为：

- `screener-scheme-panel.tsx`
- `screener-run-panel.tsx`
- `screener-result-panel.tsx`
- `screener-review-panel.tsx`

保留一个顶层 `screener-workspace.tsx` 做页面状态编排。

## 6. 状态管理边界

前端状态至少分成四类：

1. 当前选中方案
2. 当前运行状态
3. 当前结果集合
4. 当前方案反馈统计

禁止把所有状态继续堆在一个巨大组件里。

## 7. API 消费约束

前端只能消费后端显式提供的：

- 方案列表
- 方案详情
- 版本摘要
- 运行状态
- 结果列表
- 方案统计

禁止前端自行推断：

- 方案版本逻辑
- 哈希逻辑
- 质量门控判定规则
- 默认 bucket 规则

## 8. 第一阶段允许的妥协

可以接受：

- 方案创建/编辑先不放在主页面，先只做选择与运行
- 复盘区先只做只读统计摘要
- 结果区先展示组级因子解释，不强求原子因子全展开

不建议接受：

- 页面里看不到方案版本
- 运行完成后看不到本次方案摘要
- 结果区继续只显示 `rule_version`

## 9. 与后端的依赖顺序

前端开发必须建立在以下后端能力已存在的前提上：

1. 方案列表接口
2. 方案版本读取接口
3. workflow run 可绑定方案
4. run detail 返回方案摘要
5. 方案 stats 接口

如果这些接口未落地，不应先写完整交互。

## 10. 验收标准

第一阶段前端完成的标准：

1. 用户能清楚知道当前运行的是哪套方案
2. 用户能切换方案并发起运行
3. 运行结果页能显示方案版本和关键摘要
4. 单只结果详情不再脱离方案上下文
5. 页面能查看某方案的历史运行和基础反馈统计
6. 页面结构清晰分为方案、运行、结果、复盘四区

## 11. 对 Codex 的实现提醒

如果进入前端编码：

- 先从 API types 补齐开始
- 再拆组件
- 再接方案区与运行区
- 最后再接结果区和复盘区

不要先写复杂视觉交互，再回头补数据模型。
